# OpenMCP 重磅推出调用链可视化 & 季度小结

## 前言

在大约一个月前，我当时预热了 openmcp 近日在开发的新功能：[openmcp 新功能预告：工具调用链可视化](https://linux.do/t/topic/967171)。

中间发生了超多事情，忙到爆炸，所以也是暂时把 0.1.13 的发布进度延缓了，最近干掉了一堆事情后，也是终于能发布了。

> 因为我在开发的 QQ MCP 里急需里面一些新功能，0.1.13 于是变为了我的计划上的一条关键路径，让我不得不放下手里的活来赶紧发布这个我自己非常需要的版本。

所以，这个版本更新了啥呢？我说说吧。

* * *

## Update

### 1\. 增加工具调用链的可视化

我们在测试 mcp 时，往往是通过 prompt 模板和若干组预设输入查看它的多种指标是否达标。

![](https://pic1.zhimg.com/v2-2eacf95700c2e49a443140c669129b2e_r.jpg?source=d16d100b)

但是纷繁复杂的文字看多了，总是损伤理智，所以我提供了三个模态的调用链可视化，来快速查看每一次调试的性能总结，点击左上角「单模型对话」的按钮，点击下拉列表的第三个「工具链调用」即可，三个模态的效果分别如下：

\\  
表格模式：通过表格的形式，从上到下展示节点类型、输入输出、耗时等等信息。

![](https://pic1.zhimg.com/v2-5b33d3819e5796d8d86b004fe4c107f7_r.jpg?source=d16d100b)

流程图模式：通过流程图来展示每一个步骤后续是如何的，对于固定和构建工作流来说，这个步骤会有一定的辅助作用。

![](https://pica.zhimg.com/v2-a0cae1023f8bb38f557eb6e3fdbee78e_r.jpg?source=d16d100b)

有限状态机模式：这个模式主要用于展示在当前大模型下，不同工具在单次任务中的依赖性如何。孤立的、没有出度入度的节点就是没有使用的节点。这个功能可以帮助你来原子化地迭代当前的 MCP。

![](https://picx.zhimg.com/v2-3efc9b4488231a5a0303d8b53e87dfc7_r.jpg?source=d16d100b)

### 2\. 简化工具测试的数组型表单组件

这是对我过去的一个设计做减法了，我们都知道，MCP 协议中，允许函数的参数不仅仅是基础数据类型，也可以是复杂对象，而如何针对复杂对象设计他们的元表单一直是前几个版本在做的事情。

我过去的做法是：把数组类型数据做一个类似 el-tag-input 的表单组件，对象类型数据则使用 json 编辑器来作为表单容器。通过三个版本的迭代，我明白了奥卡姆剃刀的妙用，所以在这个版本中，对于所有非基础类型的数据，我统一使用了 json 编辑器来作为容器：

![](https://picx.zhimg.com/v2-728579978eabd36364c1253559983151_r.jpg?source=d16d100b)

随机 mock 和 AI mock 功能都会把生成数据直接填入其中。

### 3\. bug 修复

最近因为 QQ MCP 的驱动，我们开始在 streamable http 类型的 MCP 服务器上开始了大量的测试，也发现了 stateful 和 stateless 两类连接模式下的很多问题，加紧了一波对应的修复，细节太多，所以我就一笔带过了。

除此之外，感谢某个朋友的提醒，我才发现，我们之前一直把 mock 写作 mook，这次也是修复了这个“低级错误” 🤣

## 季度小结 & 关于后面的安排

基础调试功能，我想应该告一段落了，openmcp 后续的重点会主要落实在「验证」和「数据收集」这两个环节展开。这也是我这两个月和不少大厂和一线创业公司的朋友交流下来后，达成的共识：我们认为目前的 agent 开发周期的主要痛点并不是开发，而是验证。而形成核心壁垒的则是验证的基石——验证器（evaluator）和用于回流后训练大模型的真实场景的数据。
